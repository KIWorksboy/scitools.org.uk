Collaboration

Iris and Cartopy are free and open source projects, published under the LGPLv3 licence. Both libraries work extensively with third party free and open source libraries.

One of the primary objectives of this approach is to aid collaboration and collaborative development.

Depending on SciTools

The SciTools projects provide capabilities for developers to make use of. Projects depending on SciTools Libraries are encouraged to engage with the development process, through the Community Forums and the GitHub code development sites. It is hoped that this will lead to developers contributing code to SciTools Libraries.

Contributing to SciTools

Contributions to the SciTools open source libraries are encouraged: the pages on governance and the developer's guide provide information on what is involved in SciTools collaborative development.

Small scale contributions fit well with this model, such as bug fixes and new feature implementations to current modules. Larger scale development activities require different approaches.

Developing alongside SciTools

For a larger scale activity it may be helpful to develop slightly separate from SciTools, to enable an approach to be cultivated and trials of coding methods to be conducted.

Often the results of such activities may be suggestions of code contributions to a SciTools module, useful code which does not fit with a particular SciTools module's scope and, in general, some combination of these two scenarios.

Forking

One option is to fork a SciTools module, following the path for contributing, but without an explicit intention of putting in a pull request. This enables a developer to experiment and publish the results of their experiment, under the terms and conditions of the SciTools module. Such an experimental fork can serve as a public testing ground for ideas and implementations, giving a tangible and testable code base that may help with discussions of approaches and agreement on a particular approach which could lead to a pull request on a SciTools module and code inclusion.

Publishing Code

An alternative approach is to set up another project, dependent on SciTools, and develop this project.

For any potential collaborative development activity it is important that the code base is available to collaborators and licensed to enable collaboration.

Publishing on a site such as GitHub, the SciTools Contributors approach and LGPLv3 license present one way of managing this, but it is the responsibility of a publisher to manage these matters as they see fit.

Such public projects, dependent on SciTools modules are likely to provide a rich seam of ideas and approaches, leading to developments in SciTools modules which can be adopted and shared across the SciTools communities.

Adoption of Code

Adoption of code into SciTools libraries will follow the governance and then refer to our developer's guide procedures. In most cases discussions via Github issues and the developer communities ( iris and cartopy ) will be required, to detail scope, interfaces and design considerations. Providing publicly available code which can be developed collaboratively will enable more detailed and focussed discussions of the relevant issues.
